## 1. 背景・目的

AilaB では業務上のコミュニケーションが Slack に集中しており、重要な意思決定・アイデア・ナレッジが会話の流れの中に埋もれてしまう課題がある。これらを Notion に自動蓄積し、将来的に Gemini AI を用いた社内ナレッジ検索（RAG）へ発展させることを目的として本ツールを開発する。

---

## 2. スコープ

### 2.1 フェーズ1（本要件の対象）

- Slack リアクションをトリガーに Notion データベースへ自動保存
- カテゴリ絵文字による分類（:pushpin: 一般 / :bulb: アイデア / :white_check_mark: 決定事項）
- 保存完了後にボットがスレッドへ返信（カテゴリ名 + Notion ページリンク）
- 重複保存の防止

### 2.2 フェーズ2（将来拡張・本要件の対象外）

- Gemini API を用いた社内ナレッジ検索（RAG）
- Slack スラッシュコマンド `/ask` によるナレッジへの質問
- Notion ページのベクトル埋め込み自動更新

---

## 3. システム概要

### 3.1 処理フロー

| ステップ | 内容 |
|----------|------|
| ① | ユーザーが Slack メッセージに :pushpin: / :bulb: / :white_check_mark: リアクションを付ける |
| ② | Slack Events API が `reaction_added` イベントを Vercel エンドポイントへ送信する |
| ③ | リクエスト署名を検証し、Redis で重複チェックを行う |
| ④ | Slack API（`conversations.replies`）でスレッド全文を取得する |
| ⑤ | Notion API でデータベースに新規ページを作成する |
| ⑥ | Redis に処理済みフラグを保存する（TTL: 30日） |
| ⑦ | Slack API（`chat.postMessage`）でスレッドに保存完了を返信する |

### 3.2 技術スタック

| レイヤー | 採用技術 | 備考 |
|----------|----------|------|
| フレームワーク | Next.js 14 (App Router) | 既存の請求書ボットと統一 |
| ホスティング | Vercel (Hobby プラン) | 既存環境を流用 |
| Slack SDK | `@slack/web-api` | イベント受信・メッセージ取得・返信送信 |
| Notion SDK | `@notionhq/client` | DB への書き込み |
| 重複防止 | Upstash Redis | 既存環境を流用 |
| AI（Phase 2） | Gemini API（gemini-1.5-flash） | RAG 用途で将来追加 |

## 4. 機能要件

### 4.1 ナレッジ保存機能

| No. | 機能名 | 詳細 |
|-----|--------|------|
| F-01 | リアクション検知 | :pushpin: / :bulb: / :white_check_mark: の3種のリアクションを検知する。それ以外は無視する。 |
| F-02 | スレッド全文取得 | リアクション対象メッセージのスレッド返信を全て取得し本文に含める。 |
| F-03 | カテゴリ分類 | :pushpin:=一般、:bulb:=アイデア、:white_check_mark:=決定事項 として Notion の Category に設定する。 |
| F-04 | Notion 保存 | メッセージ本文・投稿者名・チャンネル名・保存日時・カテゴリを DB に保存する。 |
| F-05 | 重複防止 | 同一メッセージへの同一リアクション追加による二重保存を防ぐ。 |
| F-06 | 保存完了通知 | Notion 保存完了後、リアクションされた元メッセージのスレッドにボットが返信する。返信内容にはカテゴリ名・Notion ページ URL を含める。 |

### 4.2 保存完了通知の仕様（F-06 詳細）

| 項目 | 仕様 |
|------|------|
| 通知タイミング | Notion ページ作成成功後、即時 |
| 返信先 | リアクションが付いたメッセージのスレッド（`thread_ts` を指定） |
| 返信形式 | Slack Block Kit（`section` + `context` ブロック） |
| 返信内容 | ✅ 保存完了メッセージ・カテゴリ名・Notion ページへのリンク |
| エラー時 | Notion 保存失敗の場合は通知しない（エラーログのみ） |

**返信メッセージ例：**
```
✅ Notion に保存しました
カテゴリ: 💡 アイデア
→ [ページを開く](https://notion.so/xxx...)
```

### 4.3 セキュリティ要件

| No. | 要件名 | 詳細 |
|-----|--------|------|
| S-01 | Slack 署名検証 | 全リクエストで `X-Slack-Signature` を検証する。タイムスタンプが5分以上古い場合は 403 を返す。 |
| S-02 | 環境変数管理 | トークン類は全て `.env.local` で管理し、コードへの直書きを禁止する。 |

## 5. Notion データベース設計

| プロパティ名 | 種別 | 説明 |
|-------------|------|------|
| Title | タイトル | メッセージ本文の先頭 80 文字 |
| Category | セレクト | 一般 / アイデア / 決定事項 |
| PostedBy | テキスト | Slack の投稿者表示名 |
| SlackChannel | テキスト | チャンネル名（例: #general） |
| SavedAt | 日付 | Notion 保存日時（ISO 8601） |
| FullText | ページ本文 | スレッドを含むメッセージ全文 |

**⚠️ セットアップ上の注意点（実装時に判明）**

- Notion の日本語環境ではタイトル列がデフォルトで **「名前」** になっている。コードは `Title` という名前で参照するため、必ず英語に改名すること。
- プロパティ名は**大文字・小文字を含め完全一致**が必要。`title` や `category` は不可。
- データベース作成後、必ずそのページの「**コネクト先**」から作成したインテグレーションを接続すること。接続しないと API からアクセスできない（`object_not_found` エラー）。

## 8. 実装タスク分解（インターン生向け）

以下の順番で実装することを推奨する。各タスクは独立してテスト可能な粒度に分割している。

| # | タスク | 目安工数 | 完了条件 |
|---|--------|----------|----------|
| 1 | `npm install` で依存パッケージをインストール | 0.1h | `node_modules` が生成される |
| 2 | Notion DB 作成・プロパティ設定・インテグレーション接続 | 0.5h | API からページが追加できる（プロパティ名を英語で作成すること） |
| 3 | `lib/notion.ts` 実装（ページ作成関数） | 1h | 単体テストで Notion にデータが入る |
| 4 | `lib/slack.ts` 署名検証・スレッド取得・返信実装 | 0.5h | 無効リクエストで 403 が返る |
| 5 | `events/route.ts` 受信エンドポイント実装 | 1h | ngrok 経由でイベントが届く |
| 6 | Redis 重複防止ロジック追加 | 0.5h | 二重リアクションで Notion が1件のみ |
| 7 | カテゴリ絵文字の分岐実装 | 0.5h | 3種の絵文字で Category が正しく設定される |
| 8 | スレッド全文取得の実装 | 1h | 返信も Notion ページに含まれる |
| 9 | 保存完了通知（スレッド返信）実装 | 0.5h | 保存後にボットがスレッドへ返信し Notion URL が含まれる |
| 10 | Vercel デプロイ・環境変数設定・本番動作確認 | 1h | 本番環境で Slack → Notion → 通知が動作する |

**合計目安: 約6.5時間**

**⚠️ Vercel デプロイ時の注意点（実装時に判明）**

- `app/api/slack/events/route.ts` に `export const dynamic = 'force-dynamic'` が必須。これがないとビルドエラーになる。
- Slack App の Request URL には必ず `/api/slack/events` まで含めること（ルートパスのみでは 405 エラー）。
- Vercel に環境変数を追加した後は **Redeploy** が必要。
- ボットをリアクションを使うチャンネルに `/invite @ボット名` で招待しないとイベントが届かない。

## 9. フェーズ2 設計メモ（Gemini AI 連携）

フェーズ1 完了後に以下の拡張を行う。本フェーズの実装は別途要件定義を作成する。

### 9.1 RAG アーキテクチャ（想定）

```
Slack リアクション（フェーズ1）
  ↓ ナレッジが Notion に蓄積される
Gemini Embedding API（text-embedding-004）
  ↓ テキストをベクトル化
Upstash Vector（既存 Upstash を流用）
  ↓ ベクトルを保存・類似検索
Gemini Flash（gemini-1.5-flash）
  ↓ 検索結果を文章化して回答生成
Slack スラッシュコマンド /ask
  ↓ ユーザーの質問受付
```

### 9.2 Gemini を選定した理由

- `gemini-1.5-flash` はコストが低く、社内ナレッジ検索程度の用途に最適
- Notion → Gemini のデータパイプラインが公式ドキュメントで整備されている
- 将来的に Google Workspace（Gmail / Google Drive）連携に発展させやすい

## 10. 受け入れ基準

| # | テスト内容 | 期待結果 |
|---|------------|----------|
| A-01 | :pushpin: リアクションを付ける | `Category=一般` で新規ページが作成される |
| A-02 | :bulb: リアクションを付ける | `Category=アイデア` で新規ページが作成される |
| A-03 | :white_check_mark: リアクションを付ける | `Category=決定事項` で新規ページが作成される |
| A-04 | スレッドありメッセージに :pushpin: を付ける | ページ本文にスレッド返信も含まれる |
| A-05 | 同一メッセージに :pushpin: を2回付ける | Notion への保存は1件のみ |
| A-06 | :tada: など対象外リアクションを付ける | 何も起きない（Notion に保存されない） |
| A-07 | 無効な署名でリクエストを送る | 403 Forbidden が返る |
| A-08 | :pushpin: リアクションを付ける | 保存完了後、元メッセージのスレッドにボットが返信する |
| A-09 | ボット返信の内容を確認する | カテゴリ名と Notion ページ URL が含まれる |
| A-10 | Notion 保存が失敗した場合 | ボット返信は送信されない |
