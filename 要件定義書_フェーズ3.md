# 要件定義書 — フェーズ3 コンテンツ拡張抽出

最終更新: 2026-03-10

---

## 1. 背景・目的

フェーズ1・2でSlackメッセージのテキスト本文のみを保存・検索する仕組みを構築した。
しかしメッセージには URL・PDF・画像などの添付コンテンツが含まれることが多く、
それらの内容がナレッジとして活用できていなかった。

フェーズ3では、メッセージに紐づく外部コンテンツのテキストを自動抽出して
Notion および Upstash Vector に追記することで、検索精度と情報量の向上を図る。

---

## 2. 対象フロー

| フロー | 対象 |
|--------|------|
| リアルタイム保存 | 絵文字リアクション時（`/api/slack/events`） |
| バッチ取り込み | 過去メッセージの一括インポート（`scripts/batch-import.ts`） |
| 既存データ遡及 | フェーズ3適用前に保存済みのNotionページ（`scripts/enrich-existing.ts`） |

---

## 3. 機能要件

### 3-1. URL テキスト抽出

| 項目 | 仕様 |
|------|------|
| 抽出対象 | Slack 生テキスト内の `<https://...>` および `<https://...|表示名>` 形式 |
| 上限 | メッセージあたり最大 3 URL |
| 取得方法 | fetch（User-Agent 付き） + cheerio による HTML 本文抽出 |
| 文字数上限 | 1 URL あたり 3,000 文字 |
| タイムアウト | 8 秒（`AbortSignal.timeout`） |
| 非 HTML は無視 | Content-Type が `text/html` 以外は空文字を返す |
| 除外ドメイン | 下記参照 |

**除外ドメイン一覧**

| ドメイン | 除外理由 |
|----------|----------|
| `app.slack.com` | Slack 内部リンク（認証必須） |
| `files.slack.com` | Slack 添付ファイル直リンク（別途ダウンロード処理で対応） |
| `twitter.com` / `x.com` / `t.co` | 認証なしアクセス不可・ボット検知リスク（後述） |

### 3-2. PDF 抽出

| 項目 | 仕様 |
|------|------|
| 対象 | Slack 添付ファイル（mimetype: `application/pdf`） |
| ダウンロード | Slack Bot Token による Bearer 認証ダウンロード（タイムアウト 30秒） |
| 処理エンジン | Google Gemini Vision（`gemini-2.5-flash`）— PDFをBase64 inline data として送信 |
| 文字数上限 | 5,000 文字 |
| pdf-parse 不採用の理由 | v2 系が `pdfjs-dist` → `@napi-rs/canvas`（ネイティブモジュール）に依存しており Vercel Lambda で動作不可 |

### 3-3. 画像抽出

| 項目 | 仕様 |
|------|------|
| 対象 | Slack 添付ファイル（mimetype: `image/*`） |
| ダウンロード | Slack Bot Token による Bearer 認証ダウンロード（タイムアウト 30秒） |
| 処理エンジン | Google Gemini Vision（`gemini-2.5-flash`）— 画像をBase64 inline data として送信 |
| 文字数上限 | 2,000 文字 |
| 抽出内容 | 図・表・文字を含む詳細な日本語説明 |

### 3-4. 処理優先順

```
① PDF 添付ファイル（優先度: 高）
② URL コンテンツ（最大3件）
③ 画像添付ファイル（優先度: 低）
```

各ステップの失敗・空コンテンツは無視し、後続処理を継続する（フォールバック設計）。

---

## 4. X（Twitter）対応に関する方針

### 現状（未対応）

X は認証なしでの HTML 取得が不可能（ログインページにリダイレクト）。
`twitter.com` / `x.com` / `t.co` は除外ドメインとして URL 抽出から除外している。

### 将来対応の方針

| 項目 | 内容 |
|------|------|
| 手段 | Playwright による headless ブラウザ操作 + ユーザーセッション Cookie |
| 実行環境 | **ローカル専用スクリプト**（Vercel Lambda では Playwright 実行不可） |
| 対象 | 既存 Notion ページのうち X URL が含まれるもの（遡及適用） |
| 懸念 | X の利用規約上のリスクあり。個人利用の範囲で運用 |
| 優先度 | 低（要件確定次第実装） |

## 5. データ移行要件

### 5-1. 既存データへのフェーズ3遡及適用（enrich-existing.ts）

- Notion DB の全ページを取得し、元の Slack メッセージを逆引き
- `buildEnrichment` を呼び出してエンリッチメントテキストを追記
- Upstash Vector の埋め込みベクトルも再生成・更新
- 進捗: `scripts/enrich-progress.json` で管理（中断再開可能）

### 5-2. 重複ページ削除（dedup-notion.ts）

- 重複判定キー: `SlackChannel + SavedAt + Title先頭30文字`
- 保持基準: Notion ページ本文文字数が最多のもの（エンリッチ済みを優先）
- 削除: Notion アーカイブ + Upstash Vector 物理削除
- `--dry-run` オプションで削除前に確認可能

---

## 6. 非機能要件

| 項目 | 要件 |
|------|------|
| Vercel 互換性 | ネイティブモジュール不使用（Gemini Vision で代替） |
| タイムアウト | URL 8秒 / ファイルダウンロード 30秒 / Vercel 関数全体 10秒以内 |
| 冪等性 | エンリッチメント失敗時も Notion 保存は継続（try/catch 分離） |
| 文字数制限 | Notion ブロック 1件あたり 1,900文字以内に分割して保存 |
