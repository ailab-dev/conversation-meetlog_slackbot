最終更新: 2026-03-10

## 1. アーキテクチャ概要

```
Slack
  │
  │ reaction_added イベント
  ▼
/api/slack/events (Next.js App Router)
  │
  ├─ fetchThreadMessages()      ← スレッド全文取得
  ├─ fetchMessageFiles()        ← 添付ファイル情報取得
  │
  ├─ buildEnrichment()          ← [フェーズ3] lib/extractor.ts
  │     ├─ extractSlackFileContent()  PDF / 画像
  │     └─ fetchUrlText()             URL HTML
  │
  ├─ createKnowledgePage()      ← lib/notion.ts（1900文字チャンク分割）
  ├─ markAsProcessed()          ← lib/redis.ts（Notion保存成功後）
  ├─ embedText()                ← lib/gemini.ts
  └─ upsertVector()             ← lib/vector.ts
```

## 2. コンポーネント設計

### 2-1. lib/extractor.ts（新規）

フェーズ3の全抽出ロジックを集約したユーティリティモジュール。

```
extractRawUrls(slackRawText)
  └─ Slack 生テキストから URL を抽出（除外ドメインフィルタ付き）

fetchUrlText(url)
  └─ fetch → cheerio で body テキスト抽出（最大3,000文字）

extractPdfText(buffer)
  └─ Gemini Vision に application/pdf として送信 → テキスト（最大5,000文字）

extractImageText(buffer, mimeType)
  └─ Gemini Vision に image/* として送信 → テキスト（最大2,000文字）

extractSlackFileContent(urlPrivateDownload, mimeType, token)
  └─ Slack private URL からダウンロード → PDF/画像を振り分け

buildEnrichment(rawSlackText, files, token)
  └─ 上記を統合：PDF → URL → 画像 の順に処理して追記テキストを返す
```

### 2-2. lib/notion.ts（修正）

`createKnowledgePage` の `children` 生成部分を修正。
fullText を 1,900 文字ずつ分割して複数の paragraph ブロックとして登録する。

**変更前（問題あり）**
```typescript
children: [
  { paragraph: { rich_text: [{ text: { content: data.fullText } }] } }
]
// → fullText が 2,000文字を超えると Notion API が validation_error
```

**変更後（修正済み）**
```typescript
children: (() => {
  const chunks = []
  for (let i = 0; i < data.fullText.length; i += 1900) {
    chunks.push(data.fullText.slice(i, i + 1900))
  }
  return chunks.map(chunk => ({ paragraph: { rich_text: [{ text: { content: chunk } }] } }))
})()
```

### 2-3. app/api/slack/events/route.ts（修正）

#### 二重返信防止

```typescript
// Slack リトライリクエストは即座に 200 返却（処理しない）
if (req.headers.get('x-slack-retry-num')) {
  return NextResponse.json({ ok: true, skipped: 'retry' })
}
```

#### Redis 登録タイミング修正

```typescript
// 修正前（問題あり）: Promise.all で Notion 失敗時も Redis が登録済みになる
const [notionResult] = await Promise.all([
  createKnowledgePage({...}),
  markAsProcessed(redisKey),  // Notion失敗でも登録される
])

// 修正後: Notion 保存成功後に Redis 登録
const notionResult = await createKnowledgePage({...})
await markAsProcessed(redisKey)
```

#### フェーズ3 統合

```typescript
const [messages, channelName, files] = await Promise.all([
  fetchThreadMessages(channelId, messageTs),
  getChannelName(channelId),
  fetchMessageFiles(channelId, messageTs),   // ← フェーズ3で追加
])

// URL/PDF/画像の追加テキスト抽出
try {
  const enrichment = await buildEnrichment(rawText, files, process.env.SLACK_BOT_TOKEN!)
  if (enrichment) fullText += enrichment
} catch (err) {
  console.error('[Extractor] enrichment error:', err)
  // 失敗しても保存は継続
}
```

## 3. データフロー詳細

### 3-1. buildEnrichment の出力フォーマット

```
[元のメッセージ本文]

[添付PDF: ファイル名.pdf]
（Gemini で抽出したPDFテキスト 最大5,000文字）

[URL: https://example.com]
（cheerio で抽出した本文テキスト 最大3,000文字）

[添付画像: 画像名.png]
（Gemini Vision で説明したテキスト 最大2,000文字）
```

### 3-2. Notion ページ構造

| プロパティ | 型 | 内容 |
|------------|-----|------|
| Title | title | rawText 先頭80文字 |
| Category | select | 一般 / アイデア / 決定事項 |
| PostedBy | rich_text | Slack 表示名 |
| SlackChannel | rich_text | チャンネル名（`#general` など） |
| SavedAt | date | ISO 8601 タイムスタンプ |
| Body（ページ本文） | paragraph blocks | fullText を1,900文字ずつ分割 |

### 3-3. Upstash Vector メタデータ

```typescript
{
  title: string         // Notion Title と同じ
  category: string      // カテゴリラベル
  channel: string       // チャンネル名
  savedAt: string       // ISO 8601
  notionUrl: string     // Notion ページ URL
  fullText: string      // fullText 先頭 1,000 文字（検索時の表示用）
}
```

## 4. スクリプト設計

### 4-1. batch-import.ts

```
1. Slack conversations.history で全メッセージ取得（ページネーション対応）
2. 対象絵文字リアクションが付いたメッセージを抽出
3. fullText 生成（本文 + スレッド返信）
4. buildEnrichment でエンリッチメント追加
5. Notion 保存 → Gemini Embed → Vector Upsert
6. 進捗を batch-progress.json に保存（中断再開可能）
```

### 4-2. enrich-existing.ts

```
1. Notion DB から全ページ取得
2. SavedAt + SlackChannel から Slack メッセージを逆引き
3. buildEnrichment を実行
4. エンリッチメントがある場合: Notion にブロック追記 + Vector 再 Upsert
5. 進捗を enrich-progress.json に保存（中断再開可能）
```

### 4-3. dedup-notion.ts

```
1. Notion DB 全ページ取得（ページ本文の文字数も計測）
2. SlackChannel + SavedAt + Title[:30] でグルーピング
3. 重複グループ内で bodyLength 最大のページを保持
4. 残りを Notion アーカイブ + Vector 物理削除
5. --dry-run で確認のみ実行可能
```

### 4-4. fix-posted-by.ts

```
1. Notion DB から PostedBy = 'Unknown' のページ全件取得
2. チャンネル名 → ID マップを構築
3. SavedAt → Slack ts に変換（±120秒 または 1日ウィンドウ）
4. conversations.history でメッセージを取得・タイトルで照合
5. Slack users.info でユーザー名取得
6. Notion PostedBy プロパティを更新
7. 進捗を fix-progress.json に保存（中断再開可能）
```

## 5. 制約事項・設計上の判断

| 制約 | 対応方針 |
|------|----------|
| Vercel Lambda でネイティブモジュール不可 | `pdf-parse` を廃止し Gemini Vision で代替 |
| Notion ブロック 1件 2,000文字制限 | 1,900文字ずつ分割して複数ブロックに保存 |
| Slack イベント API タイムアウト 3秒 | `x-slack-retry-num` で即 200 返却し重複処理を防止 |
| X のスクレイピング不可 | 除外ドメインとして登録。将来は Playwright ローカルスクリプトで対応 |
| Redis 登録と Notion 保存の整合性 | Notion 成功後に Redis 登録（逐次実行に変更） |

## 6. X（Twitter）将来対応の設計方針

### 実装方針

```
scripts/extract-x-urls.ts（ローカル専用）
  ├─ Notion DB から X URL が含まれるページを抽出
  ├─ Playwright（headless Chrome）でページ開く
  ├─ ユーザーセッション Cookie をロード（storageState）
  ├─ ツイート本文・引用 RT を取得
  └─ Notion にブロック追記 + Vector 再 Upsert
```

### 制約

- Vercel では Playwright 実行不可のため **ローカル専用**
- `npx playwright install chromium` が必要
- X の利用規約上のリスクに留意（個人・社内利用の範囲で運用）
